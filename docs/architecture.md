# Architecture

The benchmark tool is a monorepo with four layers:

1. Workload specification.
2. Language-native benchmark harnesses.
3. Neutral orchestration and result normalization.
4. Website-ready artifacts and dashboard.

## Goals

- Compare .NET, Java, and Go using current stable runtime versions.
- Make each benchmark independently reproducible.
- Capture enough machine and runtime metadata to explain result variance.
- Support local development, CI runs, and dedicated bare-metal publishing runs.
- Publish raw output and normalized summaries so readers can audit conclusions.

## Non-Goals

- Hiding ecosystem differences behind a single microbenchmark framework.
- Claiming universal language performance from a single machine or workload.
- Mixing web framework benchmarks into core runtime benchmarks without a separate
  profile.

## Components

### `specs/runtime-policy.json`

Defines release targets, optional lanes, local prerequisites, and update policy.
This is intentionally separate from benchmark definitions so runtime upgrades can
be reviewed independently from workload changes.

### `specs/benchmarks.json`

Defines canonical benchmark identities, categories, dataset sizes, correctness
checks, and execution commands for each implementation.

### Native Harnesses

Each language owns its benchmark implementation:

- .NET: BenchmarkDotNet-based project targeting `net10.0`.
- Java: JMH-based Maven project targeting JDK 26.
- Go: standard `testing.B` benchmarks targeting Go 1.26.

The harnesses should emit native tool output plus a normalized JSON file in the
schema under `schemas/result.schema.json`.

### `tools/benchctl`

`benchctl` is the neutral control plane:

- validates specs and result files;
- prints the execution plan;
- runs selected benchmark suites;
- records machine metadata;
- normalizes results for the website;
- compares result sets across commits, machines, and runtime versions.

It is written in Python using only the standard library for the initial version
so the runner does not introduce another dependency graph.

### Result Store

Results are immutable files under `results/` in this shape:

```text
results/
  raw/<run-id>/<platform>/
  normalized/<run-id>.json
  summaries/<run-id>.json
```

The website consumes normalized JSON only. Raw output remains linked for audit.

## Execution Flow

```text
benchctl plan
benchctl validate
benchctl run --platform dotnet --benchmark cpu.nbody
benchctl run --platform java --benchmark cpu.nbody
benchctl run --platform go --benchmark cpu.nbody
benchctl normalize results/raw/<run-id>
benchctl compare results/normalized/<baseline>.json results/normalized/<candidate>.json
```

## Profiles

Profiles prevent broad claims from unrelated benchmark types:

| Profile | Purpose |
| --- | --- |
| `micro` | CPU, allocation, serialization, collections, regex, hashing |
| `throughput` | Request/response, worker pools, queue processing |
| `latency` | Tail latency under controlled concurrency |
| `memory` | Allocation rate, retained heap, GC pressure |
| `startup` | Cold start, first request, CLI launch |
| `native-aot` | Optional .NET NativeAOT and Go static binary comparisons |

## Fairness Rules

- Shared input data must be byte-identical.
- Correctness checks must run before timing or inside setup.
- Runtime flags must be explicit and captured in results.
- Warmup must be long enough for JIT runtimes.
- CI results are smoke signals only; published results require a quiet machine.
- The comparison should report confidence intervals, not only a single mean.

## Website Boundary

The website should never recalculate benchmark statistics from raw tool output.
It should render already-normalized, signed or checksummed artifacts generated by
the runner.
